The apply tactic, => and : tacticals

SSReflect's apply tactic means

intro top;
first [refine top |
       refine (top _) |
       refine (top _ _) | ‚Ä¶];
clear top.

Let's see some examples of how this works.

A, B, C:Prop
A -> A


A, B, C:Prop
A -> A

apply.
Qed.

For the next example we are going to need some "bookkeeping" tacticals:

  • the => tactical moves hypotheses from the goal to the context;
  • the : tactical moves hypotheses from the context to the goal;

and also the move tactic which can be thought of as a no-op tactic for now.

Interpreting assumptions: /view

A, B, C:Prop
(A -> B) -> A -> C -> B


A, B, C:Prop
(A -> B) -> A -> C -> B


A, B, C:Prop
ab:A -> B
a:A
C -> B


A, B, C:Prop
ab:A -> B
a:A
B -> C -> B

(* the rest of the proof is trivial and can be
finished with `done` tactic *)
done.


A, B, C:Prop
(A -> B) -> A -> C -> B

(* But the pattern above is quite common so there
is a shorthand for interpreting assumptions:
`/view` *)

A, B, C:Prop
ab:A -> B
A -> C -> B


A, B, C:Prop
ab:A -> B
B -> C -> B


(* The tactics above can be combined into
`move=> ab /ab`.

But here is a recent addition to the view
mechanism: `[apply]` view which looks a bit
differently because it's actually a notation
hiding some Ltac code which is now allowed in
views. *)


A, B, C:Prop
(A -> B) -> A -> C -> B


A, B, C:Prop
B -> C -> B

done.

(* Or, more concisely: `by move/[apply].`.
Here, we've used the `by` tactical which works a
bit like `done` executed after a tactic. *)
Qed.

The by tactical takes a list of tactics (or a single tactic), applies it to the goal and then tries to finish the proof using some simple proof automation. E.g. by [] means "the proof of the goal is trivial", because Coq does not need any hints from the user. This is equivalent to using the done tactic.

The tactics/tacticals by, exact and done are called "terminators" because if these do not solve the goal completely, the user gets an error message. It is advised to put the terminators to mark where the current subgoal is proven completely. This helps fixing proofs when they break, for instance, due to changes in the definitions of the formalization.

A, B, C:Prop
(A -> B) -> (B -> C) -> A -> C


A, B, C:Prop
(A -> B) -> (B -> C) -> A -> C


A, B, C:Prop
ab:A -> B
_Hyp_:B -> C
B -> C



A, B, C:Prop
(A -> B) -> (B -> C) -> A -> C

(* Or, sometimes we might want to move an
assumption to the context just temporarily: *)

A, B, C:Prop
ab:A -> B
(B -> C) -> B -> C

done.

(* Btw, if you are a fan of providing proof terms
explicitlty, then you can do it like so: *)

A, B, C:Prop
(A -> B) -> (B -> C) -> A -> C

exact: catcomp.
Qed.

ab:A -> B
_Hyp_:A
C

_Hyp_:A
B

(* `=>` serves here as a 'and then do this'
connector *)

A, B, C:Prop
(A -> B -> C) -> (A -> B) -> A -> C

(* Here is a, perhaps, weird proof using the
`/[dup]` view lemma which duplicated the top
assumption. *)
by move=> abc ab /[dup] /ab b /abc /(_ b).
Qed.

The rewrite tactic

SSReflect overloads the rewrite tactic. And augments it with a powerful pattern language to select a place in your goal to perform rewriting.

It's much more powerful than the vanilla rewrite tactic, however, if you ever need to use the vanilla version you can do it but using the rewrite -> equation or rewrite <- equation syntax.

A, B, C:Prop
T:Type
P:T -> Prop
forall x y, x = y -> y = x


A, B, C:Prop
T:Type
P:T -> Prop
forall x y, x = y -> y = x


A, B, C:Prop
T:Type
P:T -> Prop
x, y:T
x_eq_y:x = y
y = x

(* `-` means 'rewrite from right to left' *)

A, B, C:Prop
T:Type
P:T -> Prop
x, y:T
x_eq_y:x = y
x = x

by [].


A, B, C:Prop
T:Type
P:T -> Prop
forall x y, x = y -> y = x

(* we can make it shorter by using the `->` (or `<-)
intro-pattern. *)
by move=> ?? ->.
Qed.

move=>-> corresponds to something like this move=> fresh; rewrite {}fresh, where rewrite {}eq means rewrite with eq equation and clear it from the context.

Induction

associative addn


associative addn


x, y, z:nat
x + (y + z) = x + y + z


y, z, x:nat
IHx:x + (y + z) = x + y + z
x.+1 + (y + z) = x.+1 + y + z

(* reduction for addn is blocked, so we have to
use `rewrite`: *)

y, z, x:nat
IHx:x + (y + z) = x + y + z
(x + (y + z)).+1 = (x + y + z).+1

(* `!` means 'one or more rewrites' *)
by rewrite IHx.
Qed.


commutative addn


commutative addn


x, y:nat
x + y = y + x


y, x:nat
IHx:x + y = y + x
x.+1 + y = y + x.+1

(* `first` tactical lets us to not focus on
trivial goals |*)
by rewrite addSn IHx -addSnnS.
Qed.
